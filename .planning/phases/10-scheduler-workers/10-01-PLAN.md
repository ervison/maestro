# Scheduler & Workers Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` (recommended) or `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Execute planner DAG tasks in parallel with LangGraph `Send`, domain-specialized workers, per-worker workdir enforcement, and recursion-depth safety.

**Architecture:** Keep the existing planner and `_run_agentic_loop()` intact. Add a small multi-agent execution module that compiles a `StateGraph`, reconstructs dependency readiness from `dag + completed + failed` on each scheduler pass, fans out ready tasks with `Send`, and loops workers back into the scheduler until the DAG is exhausted. Workers stay thin: validate depth/workdir, compose the domain system prompt, call `_run_agentic_loop()`, and return reducer-safe partial state updates only.

**Tech Stack:** Python 3.12, LangGraph 1.1.6 `StateGraph` + `Send`, `graphlib.TopologicalSorter`, existing `maestro.agent._run_agentic_loop`, existing `maestro.domains.get_domain_prompt`, existing `maestro.tools.resolve_path` / `execute_tool` path guard.

## Goal And Scope

This phase adds the execution half of multi-agent mode. Phase 09 already produces a validated DAG in `state["dag"]`; Phase 10 must run that DAG safely.

In scope:

1. Scheduler computes ready tasks from DAG dependencies.
2. Ready tasks are dispatched in parallel via LangGraph `Send`.
3. Workers use domain prompts and reuse `_run_agentic_loop()` unchanged.
4. Worker failures are collected without crashing unrelated branches.
5. Recursion depth is explicit and guarded.
6. Shared state updates are reducer-safe under parallel execution.

Out of scope for this phase:

1. CLI `--multi` wiring.
2. Nested recursive planning/execution beyond installing the guardrails and state shape needed for it.
3. Streaming partial multi-agent progress to the terminal.

## Research Notes

Context7 LangGraph docs confirm these implementation constraints:

1. `add_conditional_edges(..., routing_fn, ["worker"])` is the supported way to fan out `list[Send]` in Python.
2. Parallel branches overwrite plain state keys by default; reducer-backed keys must be used for any shared parallel writes.
3. Nodes in a parallel superstep complete before the graph advances, which fits a scheduler -> workers -> scheduler batch loop.
4. A routing function used for `Send` should return `list[Send]`; mixing `Send` and string edge names in the same router is error-prone and should be avoided.

That last point is the main design constraint here: termination should not be mixed into the same routing function that returns `Send`. Use a small two-step structure instead:

1. `scheduler_node` computes readiness and termination metadata.
2. `scheduler_route` returns `END` or a dedicated `dispatch` node.
3. `dispatch_route` returns `list[Send]` to `worker`.

## Proposed State Shape

Keep the existing state keys and add only the minimum execution keys needed for correctness:

1. `failed: Annotated[list[str], operator.add]`
Why: the scheduler needs task IDs for failed workers so it does not redispatch them forever.

2. `ready_tasks: list[dict]`
Why: scheduler computes the next batch once, then the dispatch router fans out that snapshot.
Note: this key is scheduler-owned and not written by workers, so it does not need a reducer.

3. `current_task_id`, `current_task_domain`, `current_task_prompt`
Why: each `Send` payload needs worker-local fields without forcing the worker to parse the whole DAG again.

4. Keep `completed`, `outputs`, and `errors` as the shared fan-in fields.
Why: `completed` already uses `operator.add`, `errors` already uses `operator.add`, and `outputs` already has a merge reducer that is sufficient for `task_id -> output` writes.

Important implementation note:

1. Do not remove or repurpose existing keys used by planner tests.
2. Prefer `NotRequired[...]` for worker-only keys so planner-only state fixtures stay small.
3. Do not add `in_progress` unless runtime behavior proves it is necessary; LangGraph supersteps already give a clean scheduler-after-batch model.

## Execution Design

### `run_multi_agent(...)`

Introduce a small public helper in `maestro/multi_agent.py` with an explicit `depth` argument and a defaulted `max_depth`:

```python
def run_multi_agent(
    task: str,
    *,
    workdir: Path,
    auto: bool,
    depth: int,
    max_depth: int = 2,
    provider=None,
    model: str | None = None,
) -> dict[str, str]:
    ...
```

Rules:

1. `depth` must not have a default.
2. `max_depth` defaults to `2`.
3. This helper is graph-facing only for now; CLI wiring can consume it later.

### `scheduler_node(state)`

Responsibilities:

1. Validate `state["dag"]` can be materialized as `AgentPlan`.
2. Rebuild a fresh `TopologicalSorter` from `plan.tasks` every time the scheduler runs.
3. Call `ts.done(task_id)` for every task in `completed` and `failed` so those tasks are treated as terminal.
4. Read `ts.get_ready()` to compute the next runnable batch.
5. Exclude already terminal task IDs as a defensive guard.
6. Set `ready_tasks` to the serialized task payloads for the dispatch phase.
7. Detect three end states:
   1. All tasks terminal (`completed + failed`) -> graph may end.
   2. No ready tasks, no unfinished tasks -> graph may end.
   3. No ready tasks, unfinished tasks remain, and some of those tasks depend on failed tasks -> graph should end with a scheduler error describing blocked tasks.

Reason for marking both `completed` and `failed` as done in the sorter:

1. Independent branches should keep moving after a worker fails.
2. Dependents of a failed task must not execute, but unrelated tasks should still complete.
3. The scheduler must be able to terminate cleanly once the remaining unfinished tasks are permanently blocked by failures.

### `scheduler_route(state)`

Return only string destinations, never `Send` objects.

Expected behavior:

1. `END` when the DAG is fully terminal or when blocked tasks have been converted into scheduler errors.
2. `"dispatch"` when `ready_tasks` is non-empty.

### `dispatch_route(state)`

Return only `list[Send]`.

Expected behavior:

1. Build one `Send("worker", {...})` per item in `ready_tasks`.
2. Each send payload includes the shared execution fields plus worker-local task fields.
3. The payload must preserve `depth`, `max_depth`, `workdir`, and `auto`.
4. The payload should not mutate `completed`, `failed`, `outputs`, or `errors` directly.

### `worker_node(state)`

Responsibilities:

1. Enforce `depth` presence and validate `depth <= max_depth` before any execution.
2. Resolve `workdir = Path(state["workdir"]).resolve()` inside the worker.
3. Compose the system prompt as `get_domain_prompt(domain)` plus the task prompt.
4. Reuse `_run_agentic_loop()` unchanged.
5. Return only partial state updates.

Successful worker return shape:

```python
{
    "completed": [task_id],
    "outputs": {task_id: output_text},
}
```

Failed worker return shape:

```python
{
    "failed": [task_id],
    "errors": [f"{task_id}: {message}"],
}
```

This is the key non-fatal behavior: worker exceptions are converted into state, not raised back into the graph for normal task failures.

### Workdir Safety Inside Worker

The project already enforces path containment in `maestro.tools`, but the worker still must enforce the workdir boundary locally because CLI-level protections are not sufficient for future recursive or programmatic entry points.

Implementation rules:

1. Always resolve the worker workdir from state, never from `Path.cwd()`.
2. Never let the worker pass a caller-controlled unresolved path directly to `_run_agentic_loop()`.
3. The worker test must prove an attempted write outside `workdir` is blocked during worker execution, not only in raw tool tests.

## Graph Wiring

Build the graph in `maestro/multi_agent.py` using this shape:

```python
builder = StateGraph(AgentState)
builder.add_node("scheduler", scheduler_node)
builder.add_node("dispatch", dispatch_node)
builder.add_node("worker", worker_node)

builder.add_edge(START, "scheduler")
builder.add_conditional_edges("scheduler", scheduler_route, ["dispatch", END])
builder.add_conditional_edges("dispatch", dispatch_route, ["worker"])
builder.add_edge("worker", "scheduler")

graph = builder.compile()
```

Notes:

1. `dispatch_node` can be a no-op that returns `{}` if needed; its purpose is to isolate the `Send`-returning router from the termination router.
2. Keep the graph compile/build logic in one module for now; splitting further is unnecessary at this size.
3. The graph runner should return the final state so callers can inspect `outputs`, `errors`, `completed`, and `failed`.

## Risks And Design Checks

1. **Risk: mixed `Send` and string routing in one function**
Mitigation: keep `scheduler_route` and `dispatch_route` separate.

2. **Risk: infinite redispatch after worker failure**
Mitigation: add `failed` task IDs and blocked-task termination logic.

3. **Risk: silent output loss under parallel fan-in**
Mitigation: keep reducer-backed shared fields and add a real two-worker graph test.

4. **Risk: worker bypasses CLI workdir guard**
Mitigation: resolve workdir inside worker and prove escape attempts fail during worker execution.

5. **Risk: accidental recursion behavior slips in without limits**
Mitigation: make `depth` explicit, install the max-depth guard now, and keep nested execution out of scope until a later phase wires it intentionally.

## Implementation Notes For The Engineer

1. Prefer one new execution module over scattering scheduler logic across `cli.py`, `agent.py`, and `planner/`.
2. Do not rewrite `_run_agentic_loop()`; this phase is explicitly about reuse.
3. Do not add new dependencies.
4. Keep the graph synchronous at the public entry point unless LangGraph forces otherwise; the current CLI/runtime is synchronous.
5. If `provider.list_models()` is needed in the new runner, mirror the already-shipped fallback pattern from `planner_node()` or `run()` instead of inventing a second model-resolution system.

2. Ready tasks fan out in parallel with `Send`.
3. Workers use domain prompts and the existing agentic loop.

5. Worker workdir containment is proven by test.
6. Recursion depth is explicit and guarded.
